# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [1.1.0] - 2026-07-05

### Added

- `--base <ref>` argument for `scan` — override the auto-detected base branch
- Automatic base-branch detection (remote default branch → `origin/main` → `origin/master` → `origin/develop` → local `main`/`master`), with a hard abort when no baseline can be resolved
- `Scanned-Commit` header in `ripple-report.md` — `--diff` incremental scans are now SHA-anchored instead of timestamp-matched (immune to rebase/amend)
- `Status` header line, separate from the cumulative severity counts, so `resolve`/`check` can update status tallies unambiguously
- Canonical finding-status vocabulary shared by all three commands — non-terminal: `OPEN`, `RESOLUTION_PLANNED`, `MITIGATED`, `WORSENED`; terminal: `RESOLVED`, `ACCEPTED_RISK`, `STALE`
- Concrete blast-radius tracing recipe (extract changed symbols → search importers → read before listing), plus non-code dependents: persisted data written by the prior version, caches spanning a deploy, in-flight messages, rolling-deploy version skew
- Rollback-safety / mixed-version-window bullets in Data Flow and Configuration & Environment
- `Confidence: low` annotation for high-impact side effects that could not be fully verified — severity is no longer downgraded for uncertainty
- `ripple-fixes.md` is a derived view of the report — regenerated by `resolve`/`check` to hold only fix plans still awaiting application, with `ripple-report.md` as the sole status authority, so `/speckit.implement` never sees finished, failed, or stale plans
- Convergence verdict in `check` — CONVERGED / NOT CONVERGED computed from findings still awaiting action, kept separate from the critical-path message so an open WARNING can't be mistaken for a clean bill
- Coverage warning in `check` when files changed since the scan weren't covered by fix-induced analysis (points to `scan --diff`)
- Compact 9-category rubric inlined in `check` so fix-induced analysis works in a standalone session
- Cross-platform prerequisite scripts via `scripts:` frontmatter (`sh` + PowerShell) and the `{SCRIPT}` placeholder, with an inline fallback for older Spec Kit CLIs that install extension commands without placeholder substitution

### Changed

- `scan` now diffs the working tree (committed + staged + unstaged + untracked) against the baseline instead of `merge-base..HEAD` — uncommitted `/speckit.implement` output is analyzed
- `check` re-verifies every non-terminal finding — previously it read only `OPEN`, so the documented scan → resolve → implement → check path verified nothing
- `resolve` queues `OPEN` and `WORSENED` findings — previously `WORSENED` findings from `check` were a dead end
- Re-scans merge into the existing report: finding IDs are globally monotonic, statuses and history sections are preserved (previously a re-scan overwrote the report)
- Fix-induced findings created by `check` now carry status `OPEN` and join the monotonic ID sequence
- `resolve --dry-run` is now guaranteed read-only (previously the workflow steps wrote files regardless)
- Coverage Gap Matrix: removed the pre-filled "N/A — single-process" row; `critical`-only scans mark unanalyzed cells "—" instead of implying a clean pass
- `Affected` field anchors on file + symbol (not just a line number) so `check` can relocate findings after fixes shift lines
- Renames/copies (`R`/`C`), type changes (`T`), binary files, and submodule bumps are handled explicitly in the change set
- Optional context now follows Spec Kit artifacts (`AVAILABLE_DOCS`, `checklists/` directory); dropped the non-existent singular `checklist.md`; `blueprint.md` is now labeled as coming from the spec-kit-blueprint companion extension
- `extension.yml` declares catalog metadata: `effect: read-write` and `category: code`
- `extension.yml` declares `effect: read-write`; `resolve` description clarified to "record a fix plan" (the command never auto-applies code changes)

### Fixed

- Hardcoded `main` base branch broke scans on `master`/`develop` repositories, silently producing an empty "clean" report
- Empty or failed diffs now abort with an explanation instead of emitting a zero-finding report
- Causation test wording excluded issues the diff *worsened* (only "introduced" was reachable); explicit zero-findings-is-valid guidance removes over-reporting pressure
- Compound category labels (e.g. "Configuration & Environment / Interface Contract") no longer allowed — one primary category per finding keeps matrix and header counts consistent

## [1.0.0] - 2026-04-17

### Added

- `speckit.ripple.scan` command — analyze implementation for untested side effects across 9 categories
- `speckit.ripple.resolve` command — interactively resolve findings with multiple fix strategies and tradeoff analysis; generates `ripple-fixes.md` for `/speckit.implement` integration
- `speckit.ripple.check` command — re-verify findings and track resolution status
- `after_implement` hook — prompt to scan after implementation completes
- 9 analysis categories: Data Flow, State & Lifecycle, Interface Contract, Resource & Performance, Concurrency, Distributed Coordination, Configuration & Environment, Error Propagation, Observability
- Severity levels: CRITICAL, WARNING, INFO
- Incremental scan support (`--diff` flag)
- Fix-induced side effect detection in check command
- Coverage Gap Matrix in report output
